## 根拠の制約
- 現在の対象コード、現在の仕様書、現在のテストケース文書だけを根拠に判断する
- `.harness/logs/` 配下、および repo 直下 `logs/` 配下の review / usage / transcript / checkpoint を根拠にしない
- `.harness/` 配下の現在タスクと無関係な補助ファイルを根拠にしない
- 過去 run の review / usage / transcript を根拠にしない
- 判定対象は「今回の対象テストケース」に限定する
- 仕様書は、今回の対象テストケースの解釈に必要な範囲でのみ使う
- テストケース文書に対象外の項目があっても、それ自体を不足指摘にしない
- 今回の対象テストケースに含まれない不足を findings に含めるのは誤りである

## 観点
- 今回の対象テストケースがテストコードでカバーされているか
- 今回の対象テストケースに対して、各検証焦点に対応するアサーションがあるか
- 今回の対象テストケースの達成に不要な独自テストを増やしていないか
- helper が仕様理解を隠していないか
  - helper 内で期待値を導出していないか
  - helper 内の分岐やループのせいでテストケース差分が本文から読めなくなっていないか
  - helper を読まないと入力条件や期待結果が分からない状態になっていないか
  - 1つのテストを理解するために複数 helper を往復する必要がないか
- setup が構造を壊していないか
  - 準備 / 実行 / 検証 の各フェーズがブロックとして明確に判別できるか
  - setup の途中の加工や中間値が多く、入力条件を追うのに手順が必要になっていないか
  - 何を準備したかが test case と結びついて読めるか
  - 実行前の時点で、このケースが何を試すのか本文から分かるか

## レビュースコープの制約
- テストケースの追加提案はしない
- 実装コードへの指摘はしない
- 今回の対象テストケースに含まれない spec 全域の不足を指摘しない
- 今回の対象テストケースを満たすために不要なテスト追加を要求しない
- 命名、コメント、美観だけの指摘はしない
- 行数が多いこと自体は問題にしない
- 単純な fixture helper、外部依存モック生成、重複する定数データの切り出し自体は問題にしない
- AAA コメントの有無自体は問題にしない。コメントがなくても構造が明確なら可
- 仕様書のスコープ外セクションに記載された項目に関するテスト不足は指摘しない

## severity の判定基準
- critical: 今回の対象テストケースの項目が完全に欠落している
- major: 今回の対象テストケースに対するアサーションが検証焦点を満たしていない、対象テストケースとの不整合がある、helper が仕様理解を隠している、または setup / AAA 構造が崩れていて入力条件や期待挙動の読み取りを妨げている
- minor: 準備 / 実行 / 検証 の境界がやや不明瞭だが、重大な誤検証には至っていない

## checklist の要件
- 今回の対象テストケースごとに 1 エントリ以上を必ず入れる
- helper に関する確認結果を 1 エントリ以上入れる
- setup / AAA 構造に関する確認結果を 1 エントリ以上入れる
- `evidence` には対象テスト名、確認したアサーション、判定根拠を具体的に書く
