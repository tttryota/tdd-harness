## 根拠の制約
- 現在の対象コード、現在の仕様書、現在のテスト実行結果だけを根拠に判断する
- `.harness/logs/` 配下、および repo 直下 `logs/` 配下の review / usage / transcript / checkpoint を根拠にしない
- `.harness/` 配下の現在タスクと無関係な補助ファイルを根拠にしない
- 過去 run の review / usage / transcript を根拠にしない

## 制約
- テストケースの範囲外の機能は実装しない
- 仕様書に記載のインターフェースに従う
- 仕様書、テストケース文書、plan、README、設定ファイルなどスコープ外ファイルは読み取り専用として扱い、変更しない
- 今回変更してよいのは current scope の実装コードと、それに対応するテストコードだけ
- スコープ外ファイルに修正が必要だと気づいても編集せず、必要なら `notes` に短く残す

## 実装原則
- 仕様書に必要な振る舞いだけを実装する
- 仕様書に `モジュール構成` がある場合は、その構成方針と責務分割に従う
- 仕様書にない新しいファイル構成やモジュール分割を、その場の判断で発明しない
- 振る舞いの理解に不要な抽象化を増やさない
- 処理の流れが追跡できる構造を保つ
- 関連ロジックを不必要に分散させない
- 将来拡張を見越した一般化を先回りしない
- その言語と既存コードベースで自然な型・表現を使う
- 既存実装を活かせる場合は、必要最小限の修正で済ませる

## 判定基準
- `decision = "noop"`:
  - 既存実装が仕様書と対象テストの両方を満たしている
  - ファイルを変更しない
- `decision = "updated"`:
  - 仕様書または対象テストを満たすために、実装の追加または修正を行った

## 出力形式
必ず以下の JSON のみを返してください。説明文や Markdown は不要です。

```json
{
  "decision": "updated",
  "why": [
    "仕様書の X 要件を満たすために処理を追加した"
  ],
  "covered_requirements": [
    "満たした仕様項目"
  ],
  "updated_requirements": [
    "今回実装・修正した仕様項目"
  ],
  "notes": []
}
```

## 出力ルール
- `why` は 1-3 件の配列にする
- `covered_requirements` には今回満たしていると判断した仕様項目を列挙する
- `updated_requirements` には今回実装・修正した仕様項目のみを入れる
- `decision = "noop"` の場合、`updated_requirements` は空配列にする
- 対応表、長文レビュー、表形式の説明は書かない
