## レビュー観点
1. DTO / Protocol 引数の振る舞い完全性
- DTO / 入出力フィールドごとに、入力値をそのまま保持するのか、strip / trim / normalize / dedupe / sort / case 変換などの変換を行うのかが明記されているか
- Protocol 引数・戻り値で使う内部型の主要フィールドと意味が揃っているか

2. 公開 API の形状明示
- 公開 API がモジュール関数かクラスメソッドか明記されているか
- DI 注入方法がコンストラクタ引数なのか、関数引数 / キーワード引数なのか明記されているか
- 主要引数名が Python 組み込み名と衝突していないか

3. モジュール構成パスと harness scope 整合性
- 仕様書のモジュール構成に書かれたパスや配置先が、上記 sourceLayout / scopePattern と矛盾していないか
- impl フローが対象ファイルを特定できない書き方になっていないか

4. 責務の分離可能性
- threading / 非同期ライフサイクル / スケジューリング責務が、ドメインロジックと未分離のまま 1 モジュールへ混在していないか
- ただし、一般論だけで広い分割提案はしない。仕様書に明示された責務境界の不足だけを指摘する

5. テストダブルのシグネチャ決定可能性
- 依存 Protocol や公開 API のメソッドシグネチャ（引数名・主要な型・戻り値）が、仕様書から一意に導出できるか
- 後続のテスト生成で `*args, **kwargs` や推測 helper に逃げないと書けない状態になっていないか

6. スコープ整合性
- requirements でスコープ外 / 不要と明示された機能が仕様書に含まれていないか
- 仕様書が requirements のスコープを超えて拡張されていないか
- スコープ外として除外すべきものを、未定義という理由で勝手に仕様追加していないか

## 指摘ルール
- 1 issue = 1 欠陥にする
- file は必ず `{{specPath}}` にする
- line は分かる範囲で書く。特定できない場合のみ null を使う
- critical は仕様書自体が読めない、または review が成立しない場合だけに使う
- 実装スタイルの好み、helper の書き方、性能一般論、コード品質一般論は指摘しない
- 「どの記述が足りないために downstream で何が決められないか」が分かる説明にする
